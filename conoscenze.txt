# Documentazione Tecnica Database: Piattaforma UJet ERP/MES
Motore Database: Microsoft SQL Server. Topology: catalogo identità e anagrafica moduli in **`[GA]`** (in alcuni ambienti legacy il nome file può comparire come CMP); ogni app operativa risiede nel proprio database silos.
Paradigma di Sicurezza: "Zero-Trust". I controlli sui permessi non sono demandati solo al codice, ma sono blindati a livello di database.

## 1. Architettura Identità: "Passaporto e Visto"
Il sistema ospita molteplici app dipartimentali mantenendo una singola base di identità, fondata sul paradigma architetturale **"1 App = 1 Database = 1 Visto"** (Bounded Context).

Ogni applicazione risiede in un **database fisico separato (silos)**, non più solo in uno schema isolato nello stesso file di dati: l'isolamento del dominio è totale. L'infrastruttura può essere **distribuita su server diversi** (es. il silos **Etichette `[ET]`** ospitato su **bcube2**, mentre altre app restano su nodi dedicati).

- **Passaporto (Auth Centralizzata):** Gestito nel database di governance identità (es. `[GA]`), tabella `[dbo].[Users]`. Il catalogo centrale (`Users`, `Modules`, `SystemConfig`, ruoli globali) resta "sacro": nessuna tabella operativa di reparto vi risiede.
- **Visto (Silos Dipartimentali):** Ogni App possiede il **proprio database**; al suo interno (tipicamente schema `[dbo]`) risiede la tabella dei permessi locali del visto (es. `[Bobine].[dbo].[Operators]`, tabelle analoghe negli altri silos). Un utente ottiene un visto per app = una riga (o equivalente) nel database di quell'app, collegata al Passaporto tramite `IDUser`.

### 1.1 Tabella `[GA].[dbo].[Users]` (Il Passaporto)
- `IDUser` (INT, PK): Identificativo assoluto (Passaporto; nel JWT è mappato tipicamente come `globalId`).
- `Name`, `Barcode` (Unique): Identificatori per il login.
- `PasswordHash`: Hash bcrypt.
- `IsActive`: Soft-delete globale.
- `DefaultModuleID` (INT, FK verso `[GA].[dbo].[Modules]`): Definisce l'app predefinita dell'utente al login (Meta-App Layer 2); il catalogo `Modules` include **`TargetDb`** per il routing verso il silos corretto.
- `ResetRequested` (BIT): Flag (0/1) che indica se l'utente ha richiesto un reset della password dal gateway. Usato per persistere l'allarme nella Captain Console fino a risoluzione.
- **Overrides di Sicurezza (Eccezioni per utente):**
  - `SessionHoursOverride` / `PwdExpiryDaysOverride`: Sovrascrivono i valori globali di SystemConfig.
  - `ForcePwdChange` (BIT): Se 1, blocca l'utente al login finché non cambia password (usato per le password temporanee del Captain).
- **Eccezioni Regole Password (Overrides):** `PwdMinLengthOverride` (INT), `PwdRequireNumberOverride` (BIT), `PwdRequireUpperOverride` (BIT), `PwdRequireSpecialOverride` (BIT). Se NULL, il sistema applica le regole globali di SystemConfig. Se valorizzati, sovrascrivono i requisiti di complessità della password per il singolo utente.
**Tracciamento di Salute Credenziali:**
  - `LastLogin` (DATETIME): Timestamp dell'ultimo login effettivo dell'utente (aggiornato da `POST /api/login`).
  - `LastBarcodeChange` (DATETIME): Timestamp dell'ultima rigenerazione del Barcode/QR Code di login (aggiornato in creazione utente e in modifica solo se il QR viene cambiato).

### 1.2 Dizionario Ruoli e Catalogo App (`[GA].[dbo].[GlobalRoles]`, `[GA].[dbo].[Modules]`, `[GA].[dbo].[AppRoles]`)
Il sistema non usa più stringhe JSON hardcodate per definire i ruoli. La sicurezza è normalizzata in Terza Forma Normale (3NF):
- `[GA].[dbo].[Modules]`: Catalogo delle applicazioni esistenti; **governa il routing** verso i silos dipartimentali tramite la colonna **`TargetDb`** (nome del database fisico di destinazione), oltre ai metadati già noti come `TargetSchema` / `TargetTable` per individuare la tabella del visto. (La colonna `RoleDefinition` JSON è deprecata).
- `[GA].[dbo].[GlobalRoles]`: È l'anagrafica aziendale centralizzata dei ruoli (es. `RoleCode`: 'Admin', 'Base', 'Master').
- `[GA].[dbo].[AppRoles]`: Tabella ponte (matrice di sicurezza) che collega un Modulo a un Ruolo Globale. Contiene le regole fisiche: `RequiresPassword`, `SessionHours`, `PwdExpiryDays`.
- `[GA].[dbo].[vw_UserAccess]`: Vista federata che **aggrega i dati da database diversi** tramite **`UNION ALL`**: ogni ramo legge la tabella visti nel silos indicato da `Modules.TargetDb`. Per ogni combinazione utente/app, calcola dinamicamente **`IDGlobalRole`** incrociando i **`RoleCode` locali** (o equivalenti mappati dal reparto) con la tabella **`GlobalRoles`**, così il login e le API ricevono un modello permessi unico senza hardcoding per app. Il backend Node.js usa questa vista (o la stessa logica materializzata) per accessi e moduli autorizzati.

### 1.3 Tabelle Dipartimentali (I Visti)
**Esempio: `[Bobine].[dbo].[Operators]` (Reparto Bobine)**
Contiene solo attributi lavorativi locali.
- `IDOperator` (INT, PK): ID storico usato nei Log.
- `IDUser` (INT, FK): Relazione con Users.
- `Admin` (BIT): Ruolo locale nel reparto.
- `StartTime` (VARCHAR): Orario turno specifico.

### 1.4 Regole per le Tabelle delle App (I Visti) e Routing Dinamico
Per garantire che il backend (`serverbobine.js`) sia dinamico quando si aggiungono nuove app, vanno rispettate queste regole d'oro (paradigma **1 App = 1 Database**):
1. **Creazione Silos:** Provisionare un **database dedicato** per la nuova app (es. `[Magazzino]` su un'istanza SQL definita), con schema operativo coerente (spesso `[dbo]`).
2. **Struttura Visto:** Nel database dell'app, creare la tabella Visti che DEVE contenere sempre: la Foreign Key verso `IDUser` (Passaporto), la colonna del Ruolo (es. `Admin` o `Role`), e la colonna `IsActive` (BIT, fondamentale per il soft-revoke dei permessi).
3. **Registrazione (Catalogo App):** Registrare la nuova App in `[GA].[dbo].[Modules]`, obbligatoriamente con **`TargetDb`** (nome del database silos), più `TargetSchema` e `TargetTable` per puntare alla tabella visti. Il backend risolve connessioni e nomi oggetto in base a questi metadati.
4. **Dichiarazione dei Ruoli:** Inserire le regole di sicurezza della nuova app in `[GA].[dbo].[AppRoles]`, collegando l'app ai ruoli esistenti in `[GA].[dbo].[GlobalRoles]`.
5. **Aggiornamento Vista Federata:** Estendere `[GA].[dbo].[vw_UserAccess]` con un nuovo ramo `UNION ALL` che interroga il silos (`TargetDb`) e mappa i permessi locali al rispettivo `IDGlobalRole`.

### 1.5 Principio di Incapsulamento dei Database Dipartimentali (Regola di Fabrizio)
Un database dipartimentale (es. `[Bobine]`, `[ET]`) deve operare come un silos autosufficiente. Il codice applicativo non deve **MAI** spargere logica di reparto con join ad hoc cross-database verso `[GA]` lato client/query inline non governate.
- **Letture da GA:** Preferire viste o contratti esposti dal silos locale (es. `[Bobine].[dbo].[vw_ext_GlobalUsers]`) ove applicabile.
- **Scritture cross-catalogo:** Incapsulare in Stored Procedure nel database competente, con parametri tipizzati e controlli di sicurezza.

### 1.6 Database `[ET]` (App Etichette)
Silos dedicato alla progettazione etichette; può risiedere su istanza/server distinti (es. **bcube2**) rispetto ad altre app.

- **`[ET].[dbo].[Operators]` (Visto locale):** Permessi nell'app Etichette. Colonne tipiche: `IDUser` (legame al Passaporto), `Admin`, `IsActive`.
- **`[ET].[dbo].[Prodotti_Padre]`:** Testate etichetta (codice padre / anagrafica di testata) con **auditing** tramite riferimento operatore (es. `IDOperator` o equivalente legato all'operatore del silos).
- **`[ET].[dbo].[Componenti_Figli]`:** Righe della distinta base (figli) collegate al padre; vincolo referenziale con **`ON DELETE CASCADE`** dal record padre in `Prodotti_Padre`, così la cancellazione della testata rimuove coerentemente le righe figlie.
- **Logica incapsulata:** L'inserimento/aggiornamento orchestrato lato silos avviene tramite Stored Procedure, in particolare **`sp_SalvaEtichetta`**, che accetta tra i parametri l'**`IDUser` globale** (identità dal JWT / Passaporto, es. `globalId`) per tracciabilità e validazione, mantenendo il dominio ET autocontenuto.

## 2. Struttura delle Tabelle di Business

### 2.1 Tabella [Bobine].[Log] (Cuore Operativo)
Registra tutte le transazioni di produzione.
- Dati: `IDMachine`, `Codart`, `Lot`, `Quantity`, `IDRoll`, `bobina_finita`.
- Tracciabilità: Legato a `IDOperator` (il Visto), non a `IDUser` (il Passaporto), per mantenere il contesto dipartimentale.
- **Sicurezza (Soft-Delete e Auditing):** Usa `Eliminato = 1`. Modifiche tracciate via `ModificatoDa`, `NumeroModifiche` e System-Versioned Temporal Tables (`ValidFrom`, `ValidTo`) interrogabili tramite `FOR SYSTEM_TIME ALL`.

### 2.2 Tabella [Bobine].[Machines]
Anagrafica dei macchinari di produzione. `IDMachine`, `Machine`, `Barcode` (per selezione rapida).

### 2.3 Tabella `[GA].[dbo].[SystemConfig]`
Dizionario Key/Value per le configurazioni dinamiche (modificabili dal Captain a runtime). 
Valori: `AdminPwdExpiryDays`, `AdminSessionHours`, `OpDeleteMaxRecords`, `OpDeleteMaxHours`. Nuove regole password: PwdMinLength, PwdRequireNumber, PwdRequireUpper, PwdRequireSpecial.
*(Per aggiungere parametri, si inserisce una nuova riga, non una nuova colonna).*

## 3. Logica di Business e Stored Procedures

### sp_DeleteLogOperatore
Questa procedura garantisce che l'eliminazione (Soft-Delete) dei log da parte di operatori standard sia sicura e limitata.
**Regole di validazione (Eseguite a livello DB):**
1. Il record deve esistere e non essere già eliminato.
2. L'operatore che richiede l'eliminazione deve corrispondere a colui che ha creato originariamente il record.
3. Il record non deve essere più vecchio del limite stabilito in SystemConfig (es. 48 ore).
4. Il record deve rientrare tra gli ultimi N record inseriti dall'operatore, stabiliti in SystemConfig (es. ultimi 2 inserimenti).
Se tutte le condizioni sono soddisfatte, esegue `UPDATE ... SET Eliminato = 1`.

## 4. Architettura Meta-App e Sicurezza Frontend

Il sistema è diviso in due Layer principali:

- **Layer 1 – Gateway di accesso**  
  Punto di ingresso unico (futuro `index.html`) che gestisce login, routing iniziale verso le app dipartimentali e orchestrazione globale.

- **Layer 2 – App operative**  
  Applicazioni pure (es. `bobine.html`) focalizzate sul dominio di reparto. Non gestiscono direttamente login e permessi globali, ma consumano le decisioni prese a monte, reagendo ai dati di sicurezza aggregati (es. oggetto globale `window.SecurityData`).

### Sicurezza Globale e Script `sicurezza.js`

La sicurezza e le autorizzazioni – inclusi i WebSocket e i redirect brutali per sessione scaduta – sono gestite da uno script globale `sicurezza.js` residente nel Layer 1:

- Valida il token JWT e le informazioni di identità/moduli autorizzati.
- Esegue redirect immediati in caso di sessione scaduta o permessi insufficienti.
- Gestisce la richiesta di cambio password obbligatorio.

Se è richiesto un cambio password, `sicurezza.js` inietta un "Sipario" (overlay DOM) sopra il Layer 2:
- Blocca l'interazione con l'app operativa.
- Può essere attivato al caricamento (se l'utente ha il flag a DB) oppure in tempo reale dal Captain via WebSockets.
- Al termine della procedura, aggiorna il JWT nei cookie e chiude il Sipario SENZA ricaricare la pagina, emettendo un evento `securityCurtainResolved`.
- REGOLA PER LO SVILUPPO DI NUOVE APP (Layer 2): Ogni nuova app deve ascoltare l'evento `document.addEventListener('securityCurtainResolved', ...)` per aggiornare il proprio stato in memoria locale (es. `state.currentOperator.forcePwdChange = false`) in modo silente, preservando il lavoro in corso dell'utente.

### Captain Console e DefaultModuleID

La **Captain Console** è l'interfaccia amministrativa che governa l'accesso alle app dipartimentali:

- Aggiunge/rimuove utenti dalle tabelle di reparto (es. `[Bobine].[Operators]`) per assegnare o revocare i \"Visti\" locali.
- Imposta `DefaultModuleID` su `[dbo].[Users]`, che definisce **l'app di destinazione principale** (Layer 2) che verrà proposta all'utente al login.
- **Manutenzione UI (moduli):** Tutte le selezioni di modulo / app di default nella Captain Console devono essere **dinamiche**: i menu a tendina sono popolati dall'API (es. **`GET /api/admin/modules`**, dati da **`[GA].[dbo].[Modules]`**) e **non devono essere hardcoded** nell'HTML.

Combinando `DefaultModuleID` con la lista di moduli autorizzati, `sicurezza.js` può:

- Calcolare il redirect iniziale post-login.
- Validare la navigazione cross-modulo nella Meta-App.
- Applicare in modo coerente le politiche di sicurezza globali su tutte le app operative.

### Backend `serverbobine.js`, pool SQL e JWT

- **Connessioni multiple:** Il server gestisce **pool `ConnectionPool` dedicati** per ciascun silos / database di lavoro, separati dalla connessione globale al catalogo identità, così da evitare conflitti di contesto e incroci accidentali tra transazioni su `[GA]` e sui database dipartimentali (es. `[Bobine]`, `[ET]`).
- **JWT e identità:** Nel token decodificato su `req.user`, conviene distinguere:
  - **`req.user.id`**: identificativo del **visto locale** nell'app corrente (es. `IDOperator` nel silos di reparto), usato per log e permessi contestuali.
  - **`req.user.globalId`**: identificativo del **Passaporto** (`IDUser` in `[GA].[dbo].[Users]`), usato per correlazione cross-app e per parametri come `IDUser` verso procedure nei silos (es. `sp_SalvaEtichetta`).

## 5. Architettura Primary Keys e Database
**Gestione Primary Keys (IDENTITY):**
Tutte le tabelle principali del sistema (`[dbo].[Users]`, `[Bobine].[Operators]`, `[Captain].[Captains]`, `[Bobine].[Log]`, `[Bobine].[Machines]`) utilizzano la proprietà `IDENTITY (1,1)` di SQL Server per l'auto-incremento nativo delle chiavi primarie. 

**Regola di Sviluppo Inviolabile:** Il codice backend (Node.js) NON DEVE MAI calcolare gli ID manualmente (assolutamente vietato l'uso di pattern come `MAX(ID) + 1`) e NON DEVE MAI includere le colonne identificative (es. `IDUser`, `IDLog`) nelle query di `INSERT`. Il database gestisce l'assegnazione atomica degli ID in totale autonomia, garantendo l'assenza di race-conditions.
Esposizione Rete e Sicurezza SSL (Gateway)
L'applicazione è esposta pubblicamente all'indirizzo https://rotoli.ujet.it sulla porta 443.
Il traffico HTTPS è gestito nativamente dal processo Node.js (serverbobine.js), che legge staticamente i certificati all'avvio dalla directory C:\Acme\certificati_ssl\.
Gestione Rinnovi: L'aggiornamento fisico dei file .pem è delegato a un client ACME esterno (es. win-acme) su sistema operativo Windows. Poiché Node.js carica i certificati in memoria tramite readFileSync solo allo startup, è imperativo che il sistema di rinnovo ACME sia configurato per eseguire un riavvio forzato del processo node.exe al termine di ogni rinnovo con successo, altrimenti l'app continuerà a servire il certificato scaduto presente in cache.
