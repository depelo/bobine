================================================================================
1. CONTESTO E POSIZIONAMENTO NELL'ECOSISTEMA
================================================================================

GB2 (Gabriele 2.0) e l'app MRP (Material Requirements Planning) di U.Jet s.r.l.
Sostituisce un vecchio programma Access VBA collegato al database ERP BCube.

GB2 e integrata nell'ecosistema PortalUjet come modulo dipartimentale,
seguendo il paradigma "1 App = 1 Database = 1 Visto":
  - Database silos: [GB2] su 192.168.0.163
  - Tabella Visto: [GB2].[dbo].[Operators] (IDOperator, IDUser, Admin, IsActive)
  - Registrata in [GA].[dbo].[Modules] come IDModule = 4, TargetDb = 'GB2'
  - Ruoli: Admin (UAD = Ufficio Admin) e Base (UAC = Ufficio Acquisti)
  - ENTRAMBI i ruoli richiedono password (diverso da Bobine dove Base no)

Il Captain Console gestisce l'assegnazione dei visti (permessi) agli utenti.
Dopo l'assegnazione, l'utente vede "Gabriele 2.0" nel menu laterale.

NOTA IMPORTANTE: Il file conoscenze.txt nella root del progetto bobine contiene
le regole architetturali generali di PortalUjet (Passaporto/Visto, Zero-Trust,
pool isolati, ecc.). Quelle regole si applicano anche a GB2. Questo documento
copre solo le specificita dell'app GB2/MRP.

2. ARCHITETTURA TECNICA
================================================================================

Stack:
  - Backend: Node.js + Express, integrato nel server.js principale (porta 443)
  - Database: SQL Server 2022 su 192.168.0.163
  - Frontend: SPA vanilla (HTML + CSS + JS, NO framework)
  - PDF: PDFKit (server-side)
  - Email: Nodemailer
  - Libreria DB: mssql (node-mssql)
  - Dev locale: server.js + PM2 (stessi file di produzione)
  - Crittografia: AES-256-GCM (config/crypto.js)

Server:
  - 192.168.0.163 (PC-SVILUPPO): server dove gira l'app (server.js/PM2, porta 443).
    Ospita i database: MRP, GB2, GA, BOB, CAP, PE. Ha anche una COPIA di UJET11
    che e solo per test/sviluppo — NON contiene dati reali.
  - bcube2: server ERP di produzione. Ospita il VERO database UJET11 con i dati
    reali usati dai 40 dipendenti. L'app NON si connette direttamente a bcube2.

Database utilizzati:
  - MRP (su 192.168.0.163): database operativo PRODUZIONE. Contiene:
    - Tabelle custom: ordini_emessi
    - Stored procedures: usp_CreaOrdineFornitore, usp_AggiornaStatoInvioOrdine
    - SP di prova suffissate: usp_CreaOrdineFornitore_T{id}, ecc.
    - 21 VISTE che espongono le tabelle di UJET11 su bcube2 (create da Fabrizio)
  - GB2 (su 192.168.0.163): silos dell'app (Operators, UserPreferences, TestProfiles)
  - UJET11 (su bcube2): dati ERP BCube reali — in produzione accessibile via viste MRP
  - UJET11 (su server di prova): copia locale — in prova accessibile DIRETTAMENTE

Gestione connessioni — ARCHITETTURA POOL PER UTENTE:
  config/db-gb2.js gestisce i pool con isolamento per operatore:

  poolProd (singleton condiviso):
    - Pool fisso verso MRP su 192.168.0.163
    - MAX 30 connessioni (condiviso tra tutti gli operatori in produzione)
    - Usato anche per SP, GB2, TestProfiles (sempre su 163)
    - Creato lazy al primo utilizzo, vive per tutta la sessione del server

  userStates (Map<userId, UserState>):
    - Ogni operatore ha il suo stato: pool, profilo attivo, hasRiep, lastActive
    - Il pool di prova e INDIVIDUALE: lo switch di Pablo non impatta Marco
    - MAX 5 connessioni per pool di prova (1 operatore alla volta)
    - Pool chiuso automaticamente dopo 30 minuti di inattivita (cleanup ogni 5 min)
    - Con 8 operatori simultanei: max 1 pool prod (30 conn) + 8 pool prova (5 conn ciascuno)
      = 70 connessioni teoriche, ~20 reali. SQL Server ne gestisce migliaia.

  Funzioni principali (tutte accettano userId):
    getPoolMRP(userId)      — pool attivo per l'utente (produzione o prova)
    getPoolProd()            — SEMPRE il pool produzione (per SP, GB2, TestProfiles)
    getPoolRiep(userId)      — pool per query Riep (fallback a produzione se manca in prova)
    getActiveProfile(userId) — profilo attivo per l'utente
    isProduction(userId)     — true se l'utente e in produzione
    switchToTest(userId, p)  — crea pool prova individuale per l'utente
    switchToProduction(userId) — chiude pool prova, torna a produzione

  Le route ottengono userId da getUserId(req) e lo passano a tutte le funzioni pool.
  In dev mode (skipAuth), userId = 0 (default).

  config/db.js (principale): pool per GA, PE, ET, BOB (NON usato da GB2)
  Le rotte GB2 usano SOLO db-gb2.js, MAI db.js

ARCHITETTURA VISTE MRP (SOLO PRODUZIONE):
  Le 21 viste nel database [MRP] su 192.168.0.163 fungono da ponte trasparente
  verso il database UJET11 di PRODUZIONE su bcube2. Ogni vista ha lo stesso nome
  e le stesse colonne della tabella originale in UJET11.

  QUESTE VISTE SERVONO SOLO IN PRODUZIONE. In prova, il pool si connette
  direttamente a UJET11 sul server di prova, bypassando completamente MRP e le viste.

  LISTA COMPLETA DELLE 21 VISTE (tutte puntano a BCUBE2):
    artico, anagra, artfasi, artpro, artmaga, artdef, artdefx,
    movord, movdis, movmag, ordlist, testord,
    tabmaga, tabnuma, tabciva, tabpaga,
    listprod, zzassdisp,
    __MOTIPORK, __MMTIPORK,
    Riep (punta a BCUBE2.Analisi_scorte.dbo.Riep — tabella ETL di Fabrizio)

  REGOLA INVIOLABILE: le viste devono SEMPRE puntare a [BCUBE2].[UJET11].[dbo].*
  (cross-server). NON modificarle mai per profili di prova — la prova non usa le viste.

  1. gb2-theme.js        (prima di tutto per prevenire FOUC)

  3. gb2-app.js           (chiama MrpTheme.init(), MrpDbConfig.init(), MrpParametri.init())

7. FLUSSO MRP — LOGICA DI BUSINESS
================================================================================

Elaborazione per singolo articolo (funzione caricaMRP):
  1. Legge giacenza base da artpro (ap_esist, ap_ordin, ap_impeg)
  2. Legge ordini fornitore aperti da movord (tipork='O', serie='F')
  3. Legge ordini MRP da ordlist (tipork='O','H','Y')
  4. Legge emissioni gia fatte da ordini_emessi (sottrae da ap_ordin)
  5. Calcola progressivo per magazzino/fase con disponibilita
  6. Gestisce articoli esauriti con sostitutivo (blocchi esaurimento/sostitutivo/combinato)
  7. Restituisce array di righe tipizzate: padre, magazzino, totale, figlio, ecc.

7b. NAVIGAZIONE DAL MODALE RMP (Generati/Confermati)
================================================================================

Il modale RMP (apriModaleOrdiniRmp) mostra righe da ordlist con badge
Generato/Confermato. Le righe sono cliccabili:

  Click su riga (qualsiasi tipo) -> navigaProgressiviDaRmp(codart)
    Chiude il modale e carica i progressivi del componente cliccato.
    Replica il bottone [3] "Progressivi Componente" dell'Access (frmTreeRMPcomp).

  Click su bottone drill (solo tipo Y) -> apriDrillPadreRmp(codart, magaz, fase)
    Usa endpoint /ordini-padre-rmp (self-join su ordlist via ol_olprogr).
    DIVERSO da /ordini-padre che usa movord — gli impegni Y spesso
    esistono solo in ordlist, non in movord.
    Bottone "Indietro" riporta al modale RMP (non al dettaglio ordini).

Contesto modale (currentModalContext):
  In modalita RMP: { type: 'rmp', codart, fase }
  Il bottone Indietro controlla currentModalContext.type per sapere se
  tornare al modale RMP o al modale dettaglio ordini.

IMPORTANTE — DUE TABELLE, DUE RELAZIONI PADRE-FIGLIO:
  ordlist: ol_progr (PK) <-> ol_olprogr (FK padre) — dati MRP generati da BCube
  movord:  mo_riga/numord/serie/anno/tipork <-> mo_rigaor/numordor/serieor/annoor/tiporkor
  Il modale RMP lavora su ordlist, quindi il drill padre deve usare ordlist.
  Il modale dettaglio ordini lavora su movord, quindi usa movord.

8. FLUSSO EMISSIONE ORDINI FORNITORE
================================================================================

Passo 1 — Conferma (client-side):
  L'operatore conferma quantita, data, prezzo nel pannello decisionale.
  Dati in MrpApp.state.ordiniConfermati. NIENTE va in DB in questa fase.

Passo 2 — Emissione (POST /emetti-ordine):
  Chiama SP usp_CreaOrdineFornitore (o usp_CreaOrdineFornitore_T{id} in prova).
  La SP usa il placeholder {{UJET11_REF}} risolto al deploy:
  - Produzione: [BCUBE2].[UJET11].[dbo] (scrive su UJET11 di produzione)
  - Prova T3:   [SERVER_PROVA].[UJET11].[dbo] (scrive su UJET11 di prova)
  La SP:
  a) Parsa JSON articoli con OPENJSON
  b) Arricchisce da artico (descrizione, IVA)
  c) Calcola castelletto IVA
  d) LOCK applicativo (sp_getapplock) per serializzare numerazione
  e) Legge numord da tabnuma (fonte autoritativa BCube) e la aggiorna
  f) INSERT testord + movord su UJET11
  g) INSERT ordini_emessi (registro locale)
  h) Restituisce testata + righe per il PDF

Passo 3 — PDF generato server-side con PDFKit, ritornato come base64.

Passo 4 — UI: ri-fetch dati dal server (data-driven, mai manipolazione DOM diretta).

Passo 5 — Email (opzionale): POST /invia-ordine-email con PDF allegato.
  Usa la configurazione SMTP dell'operatore loggato (da UserPreferences).

  POST /smtp/test    — verifica connessione SMTP


================================================================================
10. GESTIONE PROFILI DB (MULTI-AMBIENTE)
================================================================================

File: config/db-gb2.js, config/crypto.js

PRODUZIONE:
  Configurazione hardcoded da .env, pool fisso MRP@192.168.0.163.
  NON modificabile da UI. Le credenziali non transitano mai via API.
  Variabili .env: DB_SERVER_MRP, DB_SERVER_UJET11, DB_NAME_MRP,
                  DB_UJET11, DB_USER_MRP, DB_PASSWORD_MRP

PROVA:
  Profili per operatore in [GB2].[dbo].[TestProfiles] (password crittate AES).
  Ogni operatore puo avere piu profili di prova su server diversi.
  Pool dinamico: si connette a UJET11 sul server di prova (accesso diretto,
  senza viste MRP). Switch a caldo: tenta connessione, se ok chiude vecchio pool.

  Chiave di crittografia: .env DB_ENCRYPTION_KEY (64 hex = 32 byte).
  NON CAMBIARE MAI dopo il primo uso — le password salvate diventano illeggibili.

SWITCH PROFILO:
  POST /db/switch-production  — torna a produzione
  POST /db/switch-test        — connette a server di prova + deploy SP suffissate

  Al switch a prova:
  1. Connessione a UJET11 sul server di prova (pool test)
  2. Deploy tabella ordini_emessi su UJET11 di prova (auto-deploy)
  3. Deploy SP suffissate su MRP@163 (es. usp_CreaOrdineFornitore_T3)
  4. Check esistenza dbo.Riep sul server di prova
  5. Se Riep manca → warning all'operatore (modale, non alert)

  Al switch a produzione:
  1. Chiude pool test
  2. Riattiva pool produzione (MRP@163)
  3. Le SP di produzione sono gia deployate (nessun re-deploy)

11. ARCHITETTURA PRODUZIONE VS PROVA
================================================================================

Questa sezione e CRITICA per capire come l'app gestisce i due ambienti.

PRODUZIONE:
  Pool          → MRP su 192.168.0.163
  Query dati    → dbo.artico, dbo.anagra, ecc. = VISTE in MRP → [BCUBE2].[UJET11]
  SP            → usp_CreaOrdineFornitore (senza suffisso) su MRP@163
                  Scrive su [BCUBE2].[UJET11].[dbo].[testord], [movord], [tabnuma]
  Riep          → vista MRP → [BCUBE2].[Analisi_scorte].[dbo].[Riep]

PROVA (profilo test ID=3, server=192.168.1.50):
  Pool          → UJET11 su 192.168.1.50 (DIRETTO, niente viste)
  Query dati    → dbo.artico, dbo.anagra, ecc. = TABELLE REALI in UJET11
  SP            → usp_CreaOrdineFornitore_T3 su MRP@163 (suffisso _T3)
                  Scrive su [192.168.1.50].[UJET11].[dbo].[testord], [movord], [tabnuma]
  Riep          → se dbo.Riep esiste in UJET11@prova: usa quella
                  se non esiste: fallback a MRP@163 (dati produzione) + avviso

PERCHE LE VISTE NON SERVONO IN PROVA:
  In produzione, UJET11 e su un server diverso (bcube2) da MRP (163).
  Le viste in MRP fanno da ponte cross-server.
  In prova, il pool si connette DIRETTAMENTE a UJET11 sul server di prova.
  "dbo.artico" diventa la tabella reale, non una vista. Il codice delle route
  e identico — cambia solo dove punta il pool.

SP CON SUFFISSO UNIVOCO:
  Ogni profilo di prova genera SP con suffisso _T{ID} su MRP@163:
  - usp_CreaOrdineFornitore_T3
  - usp_AggiornaStatoInvioOrdine_T3
  Le SP usano {{UJET11_REF}} risolto al deploy: [SERVER_PROVA].[UJET11].[dbo]
  Cosi convivono tutte su MRP senza conflitti.
  Quando un profilo viene eliminato, le SP suffissate vengono droppate (pulizia).

  Le route chiamano getSpName('usp_CreaOrdineFornitore', profile) che restituisce
  il nome con o senza suffisso in base al profilo attivo.
  Le SP vengono SEMPRE chiamate dal pool produzione (getPoolProd) perche risiedono
  su MRP@163, non sul server di prova.

RIEP (CASO SPECIALE):
  Riep e una tabella ETL in [BCUBE2].[Analisi_scorte].[dbo].[Riep] (345k righe).
  NON e una tabella UJET11, quindi non esiste automaticamente sui server di prova.
  getPoolRiep() gestisce il fallback:
  - Produzione o prova con Riep: usa il pool attivo
  - Prova senza Riep: usa il pool produzione (dati reali, sola lettura)
  Il flag testHasRiep e impostato al momento dello switch.

TABELLA ordini_emessi IN PROVA:
  La tabella ordini_emessi normalmente vive su MRP. In prova, viene auto-deployata
  su UJET11 del server di prova (l'operatore lavora tutto dentro UJET11).

POOL PER UTENTE (RISOLTO):
  Ogni operatore ha il suo pool indipendente (Map<userId, UserState>).
  Lo switch di un operatore NON impatta gli altri. Dimensionamento:
  - Pool produzione: max 30 connessioni, condiviso tra tutti
  - Pool prova: max 5 connessioni ciascuno, individuale per operatore
  - Cleanup automatico: pool inattivi chiusi dopo 30 minuti
  - Shutdown graceful: tutti i pool chiusi su SIGINT/SIGTERM

16. PRINCIPI ARCHITETTURALI
================================================================================

1. UI DATA-DRIVEN:
   Dopo qualsiasi scrittura, il frontend ri-fetcha dal server e ri-renderizza.
   MAI manipolare il DOM in risposta a un evento di scrittura.

2. UN SOLO CODICE PER DEV E PRODUZIONE:
   gb2Routes.js e lo STESSO file usato in produzione e in sviluppo locale.
   NON creare backend separati per lo sviluppo.

3. CONFIG SEPARATO DAL CODICE:
   Produzione da .env (hardcoded, non modificabile da UI).
   Profili di prova per operatore in [GB2].[dbo].[TestProfiles].
   SMTP per operatore in [GB2].[dbo].[UserPreferences].

4. SP SEMPRE SU MRP@163:
   Le stored procedures vivono sempre nel database MRP su 192.168.0.163.
   In produzione: nomi senza suffisso (usp_CreaOrdineFornitore).
   In prova: nomi con suffisso _T{id} (usp_CreaOrdineFornitore_T3).
   Le route le chiamano SEMPRE via pool produzione (getPoolProd).

5. PROVA = ACCESSO DIRETTO A UJET11:
   Il pool di prova si connette a UJET11 sul server di prova.
   Le viste MRP NON sono coinvolte. "dbo.artico" e la tabella reale.

6. CRITTOGRAFIA PASSWORD:
   Password profili di prova e SMTP crittate AES-256-GCM (config/crypto.js).
   Chiave in .env (DB_ENCRYPTION_KEY). NON cambiarla dopo il primo uso.

7. SMTP PER OPERATORE:
   Ogni operatore configura il proprio SMTP in UserPreferences.
   L'email dell'ordine parte dall'email dell'operatore che la invia.
