================================================================================
1. CONTESTO E POSIZIONAMENTO NELL'ECOSISTEMA
================================================================================

GB2 (Gabriele 2.0) e l'app MRP (Material Requirements Planning) di U.Jet s.r.l.
Sostituisce un vecchio programma Access VBA collegato al database ERP BCube.

GB2 e integrata nell'ecosistema PortalUjet come modulo dipartimentale,
seguendo il paradigma "1 App = 1 Database = 1 Visto":
  - Database silos: [GB2] su 192.168.0.163
  - Tabella Visto: [GB2].[dbo].[Operators] (IDOperator, IDUser, Admin, IsActive)
  - Registrata in [GA].[dbo].[Modules] come IDModule = 4, TargetDb = 'GB2'
  - Ruoli: Admin (UAD = Ufficio Admin) e Base (UAC = Ufficio Acquisti)
  - ENTRAMBI i ruoli richiedono password (diverso da Bobine dove Base no)

Il Captain Console gestisce l'assegnazione dei visti (permessi) agli utenti.
Dopo l'assegnazione, l'utente vede "Gabriele 2.0" nel menu laterale.

NOTA IMPORTANTE: Il file conoscenze.txt nella root del progetto bobine contiene
le regole architetturali generali di PortalUjet (Passaporto/Visto, Zero-Trust,
pool isolati, ecc.). Quelle regole si applicano anche a GB2. Questo documento
copre solo le specificita dell'app GB2/MRP.


================================================================================
2. ARCHITETTURA TECNICA
================================================================================

Stack:
  - Backend: Node.js + Express, integrato nel server.js principale (porta 443)
  - Database: SQL Server 2022
  - Frontend: SPA vanilla (HTML + CSS + JS, NO framework)
  - PDF: pdfmake 0.1.72 (layout dichiarativo a tabelle, stile Crystal Reports)
  - Email: Nodemailer (SMTP per operatore, futuro: Microsoft Graph API)
  - Libreria DB: mssql (node-mssql)
  - Dev locale: server.js + PM2 (stessi file di produzione)
  - Crittografia: AES-256-GCM (config/crypto.js)

Server:
  - 192.168.0.163: server dove gira l'app (server.js/PM2, porta 443).
    Ospita i database propri dell'app: MRP, GB2, GA, BOB, CAP, PE.
    Ospita anche una COPIA di UJET11 (usabile come profilo prova).
  - BCUBE2: server BCube di produzione. Ospita il database UJET11 con i dati
    reali usati dai 40 dipendenti.
  - Server di prova: qualsiasi server con un database UJET11 (163, o altri
    configurati dall'operatore nel profilo prova).

--- PRINCIPIO FONDAMENTALE: CONNESSIONI DIRETTE, ZERO VISTE ---

  L'app si connette SEMPRE direttamente al server dove vive UJET11.
  NON usa viste, NON usa linked server, NON usa intermediari.

  L'operatore sceglie l'ambiente (produzione o profilo prova).
  L'ambiente definisce SU QUALE SERVER si trova UJET11.
  Il pool si connette direttamente a quel server.
  Le tabelle hanno gli stessi nomi ovunque (dbo.testord, dbo.anagra, ecc.).
  Il codice non cambia — cambia solo il server a cui il pool si connette.

--- MAPPA DEI DATABASE ---

  A) TABELLE BCUBE (database UJET11):
     artico, anagra, movord, ordlist, testord, artfasi, tabpaga, tabport,
     tabvalu, tabnuma, artpro, movmag, destdiv, codarfo, keyord, ecc.

     Dove vivono: sul server scelto dall'operatore.
     - Produzione → BCUBE2
     - Prova → il server configurato nel profilo (puo essere 163 o qualunque altro)

     Come raggiungerle: getPoolERP(userId)
     - In produzione: poolBcube (connessione diretta a BCUBE2/UJET11)
     - In prova: getPoolMRP(uid) (connessione diretta a server_prova/UJET11)
     In entrambi i casi e una CONNESSIONE DIRETTA al database UJET11.

  B) TABELLE DELL'APP (database MRP e GB2 su 163):
     Stanno SEMPRE su 192.168.0.163, indipendentemente dall'ambiente.
     Non cambiano mai server. Si raggiungono SEMPRE con getPoolProd().

     Database MRP@163:
       ordini_emessi  — registro ordini emessi da GB2

     Database GB2@163 (naming cross-database [GB2].[dbo].[...]):
       Operators, UserPreferences, TestProfiles, ElaborazioniMRP,
       SnapshotProposte, EmailTemplates, EmailTemplateAssegnazioni,
       EmailDrafts, DeployVersion, ordini_confermati_pending

  C) STORED PROCEDURES (database [GB2_SP] sul server di destinazione):
     Le SP vivono nel database [GB2_SP] sullo STESSO server di UJET11.
     Referenziano [UJET11].[dbo] come cross-database locale — zero MSDTC.
     Deploy automatico al boot con versioning (DEPLOY_VERSION in helpers.js).
     SP presenti:
       usp_CreaOrdineFornitore        — creazione ordine atomico
       usp_AggiornaStatoInvioOrdine   — aggiornamento stato invio email
       usp_AggiungiRigheOrdineFornitore — merge righe in ordine esistente
                                         (duplicati codart consentiti: ogni riga e' indipendente)
       usp_RimuoviRigaOrdineFornitore — rimozione singola riga + storno saldi BCube
                                         (se ultima riga → cancella ordine + ripristina tabnuma)

--- I 4 POOL E QUANDO USARLI ---

  getPoolERP(userId) — TABELLE BCUBE (lettura e scrittura)
    Connessione diretta al server dove vive UJET11.
    Produzione: poolBcube (BCUBE2 diretto, singleton, 10 conn).
    Prova: getPoolMRP(uid) (server prova diretto).
    USARE PER: qualsiasi query su testord, anagra, movord, ordlist, artico,
    artfasi, tabpaga, tabport, tabvalu, tabnuma, artpro, destdiv, codarfo,
    keyord, movmag, e qualsiasi altra tabella di UJET11.

  getPoolProd() — TABELLE APP (sempre su 163)
    Connessione a MRP@163. NON cambia con l'ambiente.
    USARE PER: ordini_emessi, [GB2].[dbo].[...] (ElaborazioniMRP,
    SnapshotProposte, UserPreferences, EmailTemplates, ecc.)
    NON USARE PER: tabelle BCube (testord, anagra, ecc.)

  getPoolMRP(userId) — pool utente (segue l'ambiente)
    Produzione: restituisce poolProd (163/MRP).
    Prova: restituisce il pool diretto al server prova.
    NON usare per tabelle BCube — in produzione punta a 163 dove
    le tabelle BCube non esistono. Per BCube usare SEMPRE getPoolERP.

  getPoolRiep(userId) — tabella Riep (caso speciale)
    Riep e una tabella ETL di Fabrizio in Analisi_scorte su BCUBE2.
    In MRP@163 esiste una vista dbo.Riep che punta a quella tabella.
    Produzione: getPoolMRP → poolProd → vista Riep su 163.
    Prova con Riep locale: pool prova diretto.
    Prova senza Riep: fallback a getPoolProd (dati reali, sola lettura).
    NOTA: Riep e l'unica vista rimasta in MRP@163. Non e una tabella UJET11.

--- REGOLA DECISIONALE PER SCEGLIERE IL POOL (CRITICA) ---

  Quando scrivi una nuova query, chiediti:
  "Questa tabella fa parte di UJET11 (BCube) o dell'app (163)?"

  TABELLA BCUBE (testord, anagra, artico, movord, ordlist, ecc.)
    → usa getPoolERP(userId)
    Connessione diretta al server dell'ambiente corrente.

  TABELLA APP (ordini_emessi, [GB2].[dbo].[...], ecc.)
    → usa getPoolProd()
    Sempre su 163, non cambia con l'ambiente.

  QUERY MISTA (dati BCube + dati app)
    NON fare JOIN cross-server. Fare DUE QUERY SEPARATE:
    - Query 1: dati BCube via getPoolERP(userId)
    - Query 2: dati app via getPoolProd()
    - Merge in Node.js per chiave (codice fornitore, codice articolo, ecc.)

  ERRORI DA EVITARE:
    ✗ getPoolProd() per tabelle BCube → su 163 quelle tabelle NON ESISTONO
    ✗ getPoolMRP() per tabelle BCube in produzione → stesso problema (punta a 163)
    ✗ JOIN tra tabelle BCube e tabelle app in una sola query → server diversi

--- SCRITTURA SU TABELLE BCUBE ---

  Per SCRIVERE su tabelle UJET11 (UPDATE anagra, ALTER TABLE, ecc.):
  - In prova: getPoolERP(userId) → diretto, funziona
  - In produzione: poolBcube o connessione diretta temporanea a BCUBE2/UJET11

  Le SP BCube (bussp_bsorgsor9_fcancella, bussp_bsorgsor9_faggiorn2) si chiamano
  tramite getPoolERP perche vivono su UJET11 dello stesso server.

--- PRINCIPI DI PERFORMANCE — PARALLELISMO E RENDERING IMMEDIATO ---

  REGOLA 1: PARALLELIZZARE TUTTO IL PARALLELIZZABILE
    Quando una route fa piu query indipendenti, lanciarle con Promise.all.
    Non aspettare che una finisca per lanciare la successiva se non c'e dipendenza.

    Esempi gia implementati:
    - caricaMRP(): 3 query parallele (artpro, ordini_emessi, ordlist)
    - GET /proposta-ordini: 3 query parallele (ordlist+JOIN, fingerprint, emissioni)
    - GET /progressivi: caricaMRP padre + query figli in parallelo
    - Frontend: caricaProposta() lancia fetch dati + caricaTemplateEmail in parallelo

  REGOLA 2: RENDERING IMMEDIATO CON SKELETON
    Quando l'utente clicca un'azione che richiede dati dal server:
    1. Switchare SUBITO alla vista target con skeleton animato (placeholder pulsante)
    2. Lanciare il fetch in background
    3. Quando i dati arrivano, sostituire lo skeleton con il contenuto reale
    4. Se errore, tornare alla vista precedente con messaggio

    L'utente percepisce reattivita istantanea — il click produce un effetto visivo
    immediato. L'attesa di 1-2 secondi passa mentre guarda la struttura della pagina.

    Esempi gia implementati:
    - Esegui Vista Progressivi: switch immediato + skeleton, dati arrivano dopo
    - Gestione Lista Ordini: skeleton con barre pulsanti durante caricamento proposte

  REGOLA 3: NON BLOCCARE IL RENDER CON DATI NON ESSENZIALI
    Se il render ha bisogno di dati A ma anche di dati B opzionali:
    - Renderizza subito con dati A
    - Carica dati B in background e aggiorna il render quando arrivano

    Esempio: template email caricati in parallelo col fetch proposte (non dopo).
    Esempio: ultimo ordine fornitore caricato lazy nel pannello espanso (non alla lista).

  REGOLA 4: DEPLOY E OPERAZIONI LENTE IN BACKGROUND
    Operazioni che l'utente non deve attendere (deploy SP, check colonne, ecc.)
    vanno lanciate fire-and-forget dopo aver risposto alla richiesta HTTP.

    Esempio: switch profilo prova → risponde subito, deploy SP in background.

--- GESTIONE CONNESSIONI — POOL PER UTENTE ---

  config/db-gb2.js gestisce i pool con isolamento per operatore:

  poolProd (singleton condiviso):
    - Pool fisso verso MRP su 192.168.0.163
    - MAX 30 connessioni (condiviso tra tutti gli operatori in produzione)
    - Usato anche per SP, GB2, TestProfiles (sempre su 163)
    - Creato lazy al primo utilizzo, vive per tutta la sessione del server

  userStates (Map<userId, UserState>):
    - Ogni operatore ha il suo stato: pool, profilo attivo, hasRiep, lastActive
    - Il pool di prova e INDIVIDUALE: lo switch di Pablo non impatta Marco
    - MAX 5 connessioni per pool di prova (1 operatore alla volta)
    - Pool chiuso automaticamente dopo 30 minuti di inattivita (cleanup ogni 5 min)
    - Con 8 operatori simultanei: max 1 pool prod (30 conn) + 8 pool prova (5 conn ciascuno)

  Funzioni esportate (tutte accettano userId):
    getPoolERP(userId)       — pool TABELLE BCube: connessione diretta a UJET11
                               Produzione: poolBcube (BCUBE2 diretto)
                               Prova: getPoolMRP(uid) (server prova diretto)
    getPoolProd()            — pool TABELLE APP: MRP@163, GB2@163 (SEMPRE lo stesso)
    getPoolMRP(userId)       — pool utente (produzione=poolProd su 163, prova=server prova diretto)
                               NON usare per tabelle BCube. In produzione punta a 163
                               dove le tabelle BCube non esistono. Per BCube: getPoolERP.
    getPoolRiep(userId)      — pool per Riep con fallback a produzione se manca in prova
    getActiveProfile(userId) — profilo attivo per l'utente
    isProduction(userId)     — true se l'utente e in produzione
    switchToTest(userId, p)  — crea pool prova individuale per l'utente
    switchToProduction(userId) — chiude pool prova, torna a produzione
    getPoolBcube()           — singleton connessione diretta BCUBE2/UJET11 (solo prod)

  Le route ottengono userId da getUserId(req) e lo passano a tutte le funzioni pool.
  getUserId legge req.user.globalId (JWT payload) — corrisponde a GA.Users.IDUser.

  config/db.js (principale): pool per GA, PE, ET, BOB (NON usato da GB2)
  Le rotte GB2 usano SOLO db-gb2.js, MAI db.js

  Per i dettagli delle colonne, consultare conoscenze.txt nella root.

--- DATABASE MRP (su 192.168.0.163 — tabelle e dati dell'app) ---

  ordini_emessi         Registro emissioni ordini dal MRP web
    id                  PK identity
    ol_progr            FK verso ordlist
    ol_tipork, ol_codart, ol_conto, ol_quant, ol_fase, ol_magaz
    ord_anno, ord_serie, ord_numord, ord_riga
    quantita_ordinata   Quantita effettivamente ordinata
    elaborazione_id     Identificativo elaborazione MRP (ID intero come stringa)
    data_emissione, operatore
    email_inviata       BIT — 1 se email inviata al fornitore (v2)
    email_inviata_il    DATETIME — timestamp invio email (v2)
    ambiente            VARCHAR(20) — 'produzione' o 'prova' (v3, default 'produzione')
                        Indice IX_ordini_emessi_ambiente per filtro ambiente
    NOTA: al primo deploy di ambiente, tutti gli ordini esistenti vengono marcati 'prova'
    REGOLA: tutte le query su ordini_emessi usano SEMPRE getPoolProd(),
            perche la tabella risiede su MRP@163. Node.js la alimenta dopo il successo della SP.

--- DATABASE GB2 (silos app su 192.168.0.163) ---

  Operators             Tabella Visto (permessi)
    IDOperator          PK identity
    IDUser              FK verso [GA].[dbo].[Users]
    Admin               BIT (1=UAD/Admin, 0=UAC/Base)
    IsActive            BIT (default 1, soft-delete)

  UserPreferences       Preferenze utente (colori, label, SMTP)
    IDUser              PK (stesso di GA.Users.IDUser)
    ColorPreset         VARCHAR(50) — 'default'|'deuteranopia'|'protanopia'|'high-contrast'|'custom'
    CustomColors        NVARCHAR(MAX) — JSON con override colori CSS
    CustomLabels        NVARCHAR(MAX) — JSON con nomi personalizzati delle variabili
    SmtpHost            VARCHAR(100) — server SMTP dell'operatore
    SmtpPort            INT — porta (default 587)
    SmtpSecure          BIT — SSL on/off
    SmtpUser            VARCHAR(100) — utente SMTP
    SmtpPassword        VARBINARY(512) — password crittata AES-256-GCM
    SmtpFromAddress     VARCHAR(255) — email mittente (Da:)
    SmtpFromName        VARCHAR(100) — nome mittente
    UpdatedAt           DATETIME

  ElaborazioniMRP       Tracciamento batch MRP (fingerprint = ol_ultagg)
    ID                  PK identity (chiave interna, usata dalle FK)
    NumeroElab          INT — contatore sequenziale PER AMBIENTE (display: "Elab. #N")
                        Non condiviso tra server — calcolato come MAX+1 per Ambiente all'INSERT.
    Fingerprint         DATETIME — data ultima elaborazione MRP (da ol_ultagg)
    RilevatoIl          DATETIME — quando l'app ha rilevato il batch
    TotaleProposte      INT — numero totale proposte nel batch
    TotaleGestite       INT — numero proposte con ordine emesso
    Ambiente            VARCHAR(20) — server destinazione (es. 'BCUBE2', '192.168.0.163')
    IDUser              INT — chi ha rilevato il batch
    Note, CreatedAt, UpdatedAt
    UNIQUE INDEX: UX_ElaborazioniMRP_Fingerprint_Ambiente (Fingerprint, Ambiente)

7. FLUSSO MRP — LOGICA DI BUSINESS
================================================================================

Elaborazione per singolo articolo (funzione caricaMRP):
  1. Legge giacenza base da artpro (ap_esist, ap_ordin, ap_impeg) — usa il pool passato come parametro
  2. Legge ordini fornitore aperti da movord (tipork='O', serie='F')
  3. Legge ordini MRP da ordlist (tipork='O','H','Y')
  4. Legge emissioni gia fatte da ordini_emessi — usa SEMPRE getPoolProd() perche'
     ordini_emessi risiede esclusivamente nel database MRP su 163, mai in UJET11 di prova.
     BUG FIX (2026-04-08): prima usava getUserId(req) dove req non esisteva nel contesto
     della funzione, causando il fallimento silenzioso della query (try/catch vuoto).
  5. Calcola progressivo per magazzino/fase con disponibilita
  6. Gestisce articoli esauriti con sostitutivo (blocchi esaurimento/sostitutivo/combinato)
  7. Restituisce array di righe tipizzate: padre, magazzino, totale, figlio, ecc.

REGOLA: caricaMRP riceve il pool come parametro (puo' essere produzione o prova per le
tabelle UJET11), ma per ordini_emessi usa SEMPRE getPoolProd() — mai il pool passato.

8. FLUSSO EMISSIONE ORDINI FORNITORE
================================================================================

Passo 1 — Conferma (client-side + safety-net DB):
  L'operatore conferma nel pannello decisionale unificato (in fondo ai progressivi).
  Il pannello e sempre tabellare (multi-riga) — anche 1 data mostra 1 riga.
  Bottone "+" per aggiungere righe manuali con date arbitrarie.
  Chiave Map: String(ol_progr) — univoca per riga ordlist.
  Dati in MrpApp.state.ordiniConfermati + persistiti via PendingSync su
  ordini_confermati_pending (safety-net: sopravvive a F5/crash/cambio PC).

  RIGHE MANUALI (ol_progr negativi):
    L'operatore puo aggiungere righe che non derivano da proposte BCube.
    Queste righe ricevono un ol_progr negativo (-1, -2, ...) come identita.
    Il counter (_manualProgrCounter in gb2-progressivi.js) decrementa da -1
    e si sincronizza con entry esistenti per evitare collisioni dopo reload.
    ol_progr < 0 = riga manuale in tutto il flusso:
    - PendingSync: parseInt("-1") = -1, funziona nativamente
    - conferma-pending.js: skip validazione SnapshotProposte per negativi
    - ordini.js: registra in ordini_emessi anche con ol_progr negativo
    - aggiornaStatoVisivo: passo 1b raccoglie entry manuali dalla Map
      (non hanno elemento DOM) e le aggiunge a confPerFornData
    - Cleanup visibileSet: esclude chiavi negative (non hanno proposta)
    - Idratazione al reload: ricostruisce entry manuali da dati pending
      senza bisogno di riga proposta in byProgr

  Il pannello decisione mostra solo righe disponibili: quelle gia confermate
  o emesse vengono filtrate (righeDisponibili) prima del rendering.

  Scorporo visivo: articoli multi-data con conferme parziali mostrano le righe
  confermate come blocchi "Ordine da emettere" raggruppati per fornitore.
  Tutte le righe del fornitore (anche con date di consegna diverse) confluiscono
  nello stesso blocco. Re-merge automatico al rimuovi conferma.

  MODELLO "1 ORDINE = 1 FORNITORE" (aprile 2026):
    L'operatore U.Jet ragiona per fornitore, non per finestra temporale.
    Tutte le righe confermate per uno stesso fornitore (anche con date di consegna
    diverse) vengono inviate in UN SOLO ordine e UN SOLO PDF.
    Le diverse date convivono come righe separate dentro movord (mo_datcons varia).
    BCube nativo fa lo stesso: vedi PDF reali (es. 28/F/2026 con 5 righe a 5 date).

    Funzioni RIMOSSE rispetto al vecchio modello "1 ordine = 1 data":
      - raggruppaPerData()       — non serve piu raggruppare per data
      - emettiOrdiniMultiData()  — niente loop, una sola call per fornitore
      - flag multiOrdine + rendering "ordini separati per data" nel modale emetti
      - logica pendingPerData (merge per data)
      - CSS: .emetti-multiordine-avviso, .emetti-gruppo-data-header,
              .emetti-gruppo-data-totale (rimossi)

  Toast post-conferma (gb2-progressivi.js confermaOrdineHandler):
    Quando l'operatore conferma righe dal pannello decisione e ne esistono GIA'
    altre per lo stesso fornitore in MrpApp.state.ordiniConfermati, mostra il
    modale di feedback "N righe aggiunte all'ordine pending del fornitore X"
    (usa MrpProposta.modale, esposto nel return del modulo).

  Logica merge con ordini esistenti (3 stati):
    1. Pending (in ordiniConfermati, no SP ancora) -> auto-merge silenzioso
       (la Map raccoglie tutte le righe con chiavi ol_progr distinte)
    2. Emesso, email NON inviata -> dialog apriDialogMergeDecision chiede
       conferma per chiamare /modifica-ordine + SP usp_AggiungiRigheOrdineFornitore
    3. Emesso, email INVIATA -> blocco con EMAIL_GIA_INVIATA, l'operatore
       deve creare ordine separato (POST /emetti-ordine)

    La SP usp_AggiungiRigheOrdineFornitore aggiunge righe IN-PLACE all'ordine
    esistente (preserva numord, ricalcola totali, rigenera PDF).
    NON ammette duplicati (codart, fase, magaz) — il dialog disabilita "Unisci"
    se rileva conflitti.

Passo 2 — Emissione (POST /emetti-ordine):
  Pool SP: in produzione usa poolBcube (BCUBE2 diretto), in prova usa getPoolMRP(uid).
  Chiama [GB2].[dbo].usp_CreaOrdineFornitore (o _T{id} in prova) sul server target.
  La SP vive nel DB [GB2] sullo stesso server di [UJET11] — tutto locale, zero MSDTC.
  La SP:
  a) Parsa JSON articoli con OPENJSON
  b) Arricchisce da [UJET11].[dbo].[artico] (descrizione, IVA) — cross-database locale
  c) Calcola castelletto IVA
  d) LOCK applicativo (sp_getapplock) per serializzare numerazione
  e) Legge numord da [UJET11].[dbo].[tabnuma] e la aggiorna
  f) INSERT testord + movord su [UJET11].[dbo]
  g) EXEC [UJET11].[dbo].bussp_bsorgsor9_faggiorn2 — aggiorna keyord+artpro (locale)
  h) Restituisce testata + righe (con ol_progr) per il PDF e per Node.js
  La SP NON scrive in ordini_emessi (vive su 163, non raggiungibile dalla SP).
  Operatore: firma 'GB2{IDUser}' (es. GB220 per utente 20)

Passo 2b — Post-emissione (dopo successo SP, in Node.js su poolProd = 163):
  a) INSERT ordini_emessi per ogni riga (con SCOPE_IDENTITY per recuperare ID)
  b) UPDATE SnapshotProposte SET Gestita=1, OrdineEmessoID=id
  c) UPDATE ElaborazioniMRP SET TotaleGestite = COUNT(...)

Passo 2c — Controllo duplicati (pre-emissione):
  POST /controlla-duplicato verifica se esistono gia ordini per stessi
  articoli+fornitore+elaborazione in ordini_emessi. Se si, mostra warning
  con opzione Procedi/Annulla.

Passo 3 — PDF generato server-side con pdfmake, ritornato come base64.

Passo 4 — UI: ri-fetch dati dal server (data-driven, mai manipolazione DOM diretta).

Passo 5 — Email (opzionale): POST /invia-ordine-email con PDF allegato.
  Usa la configurazione SMTP dell'operatore loggato (da UserPreferences).
  Dopo invio: UPDATE ordini_emessi SET email_inviata=1, email_inviata_il=GETDATE().

Passo 6 — Storico (modale):
  Bottone "Storico" nella header proposta apre modale con lista ordini emessi.
  Filtri: elaborazione corrente (checkbox), date da/a, fornitore.
  Azioni per riga: Visualizza (riapre modale risultato), PDF, Re-invia email.

9. GENERAZIONE PDF E INVIO EMAIL
================================================================================

PDF — PDFMAKE (layout dichiarativo a tabelle):
  File: utils/pdfOrdine.js
  Motore: pdfmake 0.1.72 (CommonJS, puro JS, zero dipendenze native).
  Il PDF replica il layout degli ordini BCube (Crystal Reports) usando tabelle
  dichiarative — celle con bordi chiusi automatici, come Crystal.
  Firma funzione: generaPdfOrdine(ordine, righe, options) → Promise<Buffer>
  Due layout: Italia (IT/UE) e Estero (EXTRA_UE) con note legali diverse.
  Se ambiente e "prova", il titolo mostra "PROVA" in rosso.
  Header ripetuto su ogni pagina (pdfmake headerRows).
  Assets: logo-ujet.jpeg + firma-tardioli.png (sfondo trasparente) in assets/.

  COMPONENTI PDF (formule Crystal implementate):
    - Riga tipo "D" (descrizione libera, senza codart/UM/quant/prezzo)
    - Formula @UM: Estero=unmis, Italia=ump se unmis===ump altrimenti unmis
    - Formula @QUANT: Estero=quant, Italia=quant se unmis===ump altrimenti colli
    - Prezzo 4 decimali, sconti (sc1+sc2+sc3)
    - LOT (mo_lotto se > 0)

    === LAYOUT COLONNA "COD.ARTICOLO" (sinistra) ===
    Due righe, replica BCube:
      Riga 1: movord.mo_codart             (es. "0010488")
      Riga 2: ( artico.ar_codalt )         (es. "(FET001)") — SOLO se non null/empty/'D'
    ar_codalt NON va nella colonna descrizione: è un codice identificativo alternativo/vecchio,
    sta logicamente insieme al codice principale.

    === LAYOUT COLONNA "DESCRIZIONE" (centrale) ===
    Concatenazione verticale rigorosa di 4 campi indipendenti, in quest'ordine:
      [1] movord.mo_descr     — titolo articolo principale (fontSize 8, copiato da ar_descr)
      [2] movord.mo_desint    — specifica tecnica riga ordine (fontSize 6.5, se presente)
                                Esempio: "da gr. 50/Mtq", "CON FILO RX. CNF 5 PZ..."
      [3] artico.ar_note      — note anagrafica articolo, MAX 5 RIGHE LOGICHE (split \r\n)
      [4] N.B. movord.mo_note — note riga ordine inserite dall'operatore, INTEGRALI

    Nota: mo_desint, mo_note → tabella movord (per-riga-ordine, cambiano per ogni ordine)
          ar_note, ar_codalt, ar_descr → tabella artico (anagrafica, uguali per ogni ordine
          dello stesso articolo)

    === REGOLA "MAX 5 RIGHE" DI ar_note (FONDAMENTALE) ===
    ar_note è un VARCHAR(MAX) contenente righe logiche separate da CRLF (\r\n).
    Si stampano SOLO le prime 5 righe logiche — le righe 6+ sono DELIBERATAMENTE OMESSE.

    Motivazione (verificata su BCUBE2 produzione, 15.681 articoli, 13.873 con ar_note):
      - L'operatore segue una convenzione editoriale: le prime 5 righe sono la
        "scheda tecnica pubblica" destinata al fornitore:
          Riga 1: ISTRUZIONI (cicli di lavorazione)
          Riga 2: DIMENSIONI DEL PRODOTTO FINITO
          Riga 3: CONTROLLI (procedure)
          Riga 4: TOLLERANZE DIMENSIONALI
          Riga 5: RIF. TECNICI (moduli aziendali, disegni, fustelle)
      - Dalla 6a riga in poi ci sono info INTERNE di reparto (parametri macchina GFM,
        settaggi, "CAMPO STERILE: NO", "ETICHETTA LATO OPERATORE: NO", ecc.)
        che NON devono mai arrivare al fornitore.
      - Circa il 45% degli articoli ha 6+ righe logiche, alcuni arrivano a 26.
      - Le righe sono SEMPRE ben delimitate da \r\n nel DB: l'operatore non si affida
        al wrap visivo di Crystal. La 5a riga finisce con punto/parola conclusa, mai
        tronca a metà parola.

    Implementazione (utils/pdfOrdine.js):
      const arNoteLines = ar_note.split(/\r?\n/).slice(0, 5).join('\n');

    Nota sull'edge case "wrap Crystal":
      Crystal Reports usa "Maximum number of lines = 5" che opera su righe VISIVE (post wrap).
      Se la 5a riga logica è molto lunga (>80 char), Crystal la wrappa visivamente in 2
      righe e la taglia a metà frase. Noi la stampiamo per intero (wrappata da pdfmake
      in 2 righe visive ma con testo completo). Questa differenza:
        (a) non espone info interne (quelle sono in riga 6+, mai stampata),
        (b) è a vantaggio del fornitore che riceve la riga 5 per intero,
        (c) non replica pixel-perfect il wrap Crystal ma è irrilevante ai fini informativi.

    === DISCRIMINANTE TEMPLATE PDF: VALUTA, NON PAESE FORNITORE ===
    Crystal ha due template ordini fornitore:
      - Ujetorfo.rpt → versione IT completa (D.Lgs 231, shelf-life, firma)
      - Ujetorfv.rpt → versione "valuta" semplificata (niente 231/shelf-life)

    Il nome "Ujetorfv" = "Ordini Fornitore Valuta" è un indizio: la discriminante
    per scegliere quale template usare NON è il paese del fornitore (an_nazion1)
    né HH_TipoReport, MA la VALUTA dell'ordine (td_valuta):
      - td_valuta = 0 (EUR)       → Ujetorfo.rpt (IT)
      - td_valuta != 0 (estera)   → Ujetorfv.rpt (EX)

    Prova del fuoco (verificata su BCUBE2 produzione):
      Ordine 281/F/2026, fornitore NURTEKS TEKSTIL (Istanbul, TR)
        an_nazion1 = 'TR'
        HH_TipoReport = 'EXTRA_UE'
        td_valuta = 0 (EUR)

    === BUG STORICO: NON fare .trim() PRIMA dello split su ar_note ===
    Il campo ar_note può avere righe iniziali VUOTE come convenzione editoriale
    per "spingere" le info interne fuori dalla finestra delle 5 righe pubbliche.
    Esempio reale (art. 0010966, ordine 289/F/2026):
      ar_note = "\n\n\n\n\n\n\n\nPeso Anima Cartone: 610 gr"
      (8 righe vuote + info interna su riga 9)

    Se fai .trim() PRIMA dello split, le righe vuote iniziali vengono eliminate
    e "Peso Anima Cartone" diventa riga 1 → bypassa la regola 5-righe → viene
    stampato al fornitore. MALE.

    Corretto: strip solo \r e \u00d0, NO trim sull'intero campo. Trim va fatto
    SOLO sul risultato finale (dopo split + slice(0,5) + join) per ripulire
    eventuali righe vuote che restano dopo il taglio.

    Codice corretto in buildDescrizioneRiga (pdfOrdine.js):
      const arNoteRaw = r.ar_note.toString().replace(/\r/g,'').replace(/\u00d0/g,'');
      const arNoteLines = arNoteRaw.split('\n').slice(0, 5).join('\n').trim();

    Il helper generico ss() NON va usato su ar_note perché fa .trim() — userebbe
    una "scorciatoia" che rompe la semantica della regola 5-righe.

    ar_note vs mo_note — campi SEPARATI e INDIPENDENTI:
      ar_note  = note fisse dell'articolo (anagrafica, permanenti)
      mo_note  = note operatore per quel singolo ordine (urgenze, "N.B.: Da tenere a
                 scorta presso il Vs magazzino", "ATTENDO CONFERMA PREZZI", ecc.)
      BCube NON copia ar_note in mo_note (verificato su DB: contenuti totalmente diversi).
      mo_note NON ha il limite delle 5 righe: si stampa integralmente con prefisso "N.B.:".

16. SISTEMA TRACCIAMENTO ELABORAZIONI MRP
================================================================================

L'elaborazione MRP notturna di BCube genera le proposte in ordlist. L'app GB2
deve sapere a quale "batch" appartengono le proposte correnti per:
- Distinguere proposte nuove da quelle gia gestite
- Permettere lo storico degli ordini emessi per elaborazione
- Evitare ri-emissioni accidentali

FINGERPRINT:
  Il campo ol_ultagg di ordlist contiene la data dell'ultima elaborazione MRP.
  Tutte le proposte di un batch condividono lo stesso ol_ultagg. L'app rileva
  il fingerprint con:
    SELECT TOP 1 ol_ultagg AS fingerprint, COUNT(*) AS cnt
    FROM dbo.ordlist WHERE ol_tipork='O'
    GROUP BY ol_ultagg ORDER BY cnt DESC

RILEVAZIONE (in GET /proposta-ordini):
  1. Query fingerprint (su pool MRP)
  2. Check ElaborazioniMRP per (Fingerprint, Ambiente) su poolGB2 = getPoolProd()
  3. Se non esiste: INSERT in ElaborazioniMRP + bulk INSERT in SnapshotProposte
     (chunked a 100 righe per batch, parametri $1..$N)
     Pattern INSERT-or-SELECT: catch errore duplicate key (2601/2627) → SELECT
  4. Se esiste: riconciliazione — query ordini_emessi per elaborazione_id,
     UPDATE SnapshotProposte.Gestita=1, UPDATE TotaleGestite

RISPOSTA API (cambio formato):
  GET /proposta-ordini ora restituisce:
    { elaborazione: { id, fingerprint, totaleProposte, totaleGestite }, righe: [...] }
  Il frontend estrae elaborazione e salva in MrpApp.state.

FRONTEND — 5 SCENARI UX:
  1. Blocco ri-emissione: ordine gia emesso → badge "Emesso" + bottone "Visualizza Ordine"
     Click → GET /ordine-dettaglio → riapre modale risultato con PDF ed email
  2. Storico ordini: modale con lista ordini raggruppati per documento
     Filtri: elaborazione corrente, date, fornitore. Azioni: PDF, re-invio email
  3. Warning duplicati: prima di emettere, POST /controlla-duplicato verifica
     se stessi articoli+fornitore gia emessi in questa elaborazione
  4. Badge email: icona inviata/non-inviata per ogni ordine emesso
     Testo bottone: "Invia" vs "Re-invia" in base a email_inviata
  5. Proposte gestite in verde: classe .proposta-art-gestita (bordo+sfondo verde)
     se r.emesso && r.elaborazione_id === MrpApp.state.elaborazioneId

BANNER ELABORAZIONE:
  Riga info dopo le stats: "#ID | Batch MRP: dd/mm/yyyy HH:mm | Gestite: X/Y"

CAMBIO ELABORAZIONE:
  Se il fingerprint cambia (nuova elaborazione MRP notturna), il frontend svuota
  MrpApp.state.ordiniConfermati per evitare ordini basati su proposte obsolete.

POOL — vedi regola decisionale in sezione 2. In sintesi per questa sezione:
  - ElaborazioniMRP, SnapshotProposte, ordini_emessi → getPoolProd() (dati locali 163)
  - ordlist (fingerprint) → getPoolMRP(userId) (dati ERP, segue il modo dell'utente)

AMBIENTE:
  Ogni riga in ordini_emessi ha una colonna 'ambiente' ('produzione' o 'prova').
  Le query di storico filtrano per ambiente corrente, cosi ordini di prova
  non appaiono in produzione e viceversa.
  ElaborazioniMRP ha UNIQUE index su (Fingerprint, Ambiente) per separare
  elaborazioni rilevate in produzione da quelle rilevate in prova.

FALLBACK QUERY:
  Le query su ordini_emessi usano try/catch: se le colonne email_inviata o
  ambiente non esistono ancora (pre-deploy), il catch esegue una query senza
  quelle colonne. Questo garantisce compatibilita durante il primo deploy.

25. PREZZO PER QUANTITA (perqta)
================================================================================

In BCube il prezzo di un articolo puo essere espresso "per gruppo" anziche "per pezzo".
Esempio: prezzo 31 EUR per 250 PZ → prezzo unitario = 31/250 = 0.124 EUR/PZ.

CAMPO DB: ar_perqta (artico), ol_perqta (ordlist), mo_perqta (movord)
  - Sempre compilato (mai NULL, mai 0). Valore minimo = 1 (prezzo per pezzo).
  - 197 articoli su ~15.000 hanno perqta > 1 (es. 100, 125, 250, 500, 1000).
  - Distribuzione ordini 2025-2026: ~8% delle righe hanno perqta > 1.

FORMULA VALORE: quantita * prezzo / perqta
  - Frontend: tutti i calcoli valore dividono per (perqta || 1)
  - SP usp_CreaOrdineFornitore: valore = quantita * prezzo / ISNULL(NULLIF(perqta, 0), 1)
  - SP scrive mo_perqta corretto in movord (non piu hardcodato 1.0)
  - JSON articoli inviato al backend include il campo perqta

PDF: il prezzo nel PDF e il prezzo "per gruppo" (come in BCube).
  Il totale ordine e corretto (calcolato con /perqta dalla SP).
  Crystal Reports ha una riga "Dettagli g" che mostra:
  "Attention: Unit Price is referred to [ar_un] [o_perqta]"
  quando perqta > 1. Da implementare nel nostro PDF.

NOTE NEL PDF (verificato su Crystal Reports Designer Ujetorfv.rpt + analisi DB BCUBE2):
  Crystal organizza il corpo articolo in 7 sezioni Dettagli:
    b: @CodArt | mo_descr | @UM | mo_quant | mo_prezzo | sconti | @Data
       NOTA: @CodArt è una formula Crystal che nella colonna Cod.articolo stampa
       mo_codart in riga 1 e (ar_codalt) tra parentesi in riga 2 se ar_codalt non è null.
    c: mo_desint (descrizione integrativa della riga ordine — movord)
    d: LOT mo_lotto
    e: ar_note (Can Grow=true, Maximum Number of Lines=5 — tronca il resto)
    f: N.B. mo_note
    g: Attention: Unit Price is referred to ar_un o_perqta
  Footer (Pie di pagina b):
    note/remarks: an_note + an_note2 + td_note (Crystal li stampa tutti)
    Nella nostra app: SOLO td_note. an_note/an_note2 rimossi (contatti interni).

  REGOLA "MAX 5 RIGHE" SU ar_note — ANALISI DB PRODUZIONE (BCUBE2):
  Verifica empirica su 15.681 articoli (13.873 con ar_note), campione 3.000:
    Istogramma righe logiche (split \r\n):
      1-4 righe:  1.298 articoli (43%)
      5 righe:      316 articoli (10,5%)
      6+ righe:   1.367 articoli (45,6%)   ← pattern deliberato
      Max trovato: 26 righe logiche
  Pattern editoriale dominante (usato da centinaia di articoli):
    [1] ISTRUZIONI: CICLO 1 (Ixxx) - CICLO 2 ...
    [2] DIMENSIONE DEL PRODOTTO FINITO: Lunghezza cm X - Larghezza cm Y
    [3] CONTROLLI: CICLO 1 v/Procedura 205 - CICLO 2 v/Procedura 202
    [4] TOLLERANZE DIMENSIONALI: più o meno 1 cm
    [5] RIF. TECNICI: Mod.201(Modulo Aziendale) - Sch.POL(Schema di Taglio)
    ───────── MAX 5 RIGHE PUBBLICHE → tutto ciò che segue è interno ─────────
    [6+] "*** PARAMETRI IMPOSTAZIONE GFM ***" / "CAMPO STERILE: NO"
         "ETICHETTA LATO OPERATORE: NO" / "LAVORO IN LINEA SINGOLA" / ecc.
  Conclusione:
    - Le righe sono SEMPRE ben delimitate da \r\n nel DB.
    - L'operatore NON si affida al wrap visivo Crystal per tagliare.
    - Implementazione corretta: split('\r?\n').slice(0,5).join('\n')
    - Questo replica il contenuto informativo di Crystal al 100% e protegge le
      info interne anche se pdfmake e Crystal wrappano diversamente.

ALTRI CAMPI ARTICOLO CORRELATI:
  - ar_minord: "Qta Lotto std pr/ac." (lotto standard di acquisto)
  - ar_rrfence: "RR Fence" (lead time in giorni)
  - ar_ggrior: "Giorni raggruppamento" (NON e il lead time)
  - ar_scomin: scorta minima
  Politica riordino mostra: (scorta min. X, lotto Y, lead time Z gg)

26. SISTEMA PREZZI BCUBE — TABELLA LISTINI E MOTORE PRICING
================================================================================

TABELLA LISTINI (prefisso lc_):
  Tabella BCube che contiene i prezzi con scaglioni quantita e validita temporale.
  Una riga = un prezzo per un articolo in un range quantita in un periodo di tempo.

  Colonne principali:
    lc_progr    (int)        — PK progressivo
    lc_codart   (varchar 50) — codice articolo
    lc_conto    (int)        — codice fornitore/cliente (0 = generico, >0 = specifico)
    lc_listino  (smallint)   — numero listino (FK verso tablist.tb_codlist)
    lc_prezzo   (decimal)    — prezzo
    lc_perqta   (decimal)    — prezzo per quantita (divisore), come ar_perqta
    lc_daquant  (decimal)    — quantita minima dello scaglione
    lc_aquant   (decimal)    — quantita massima dello scaglione
    lc_datagg   (datetime)   — data inizio validita
    lc_datscad  (datetime)   — data fine validita
    lc_codvalu  (smallint)   — codice valuta (0 = EUR)
    lc_codtpro  (smallint)   — codice tipo promozione
    lc_netto    (varchar 1)  — flag prezzo netto (no sconti)
    lc_unmis    (varchar 3)  — unita di misura prezzo
    lc_tipo     (varchar 1)  — tipo
    lc_codlavo  (smallint)   — codice lavorazione
    lc_fase     (smallint)   — fase
    lc_coddest  (int)        — codice destinazione

  Tabella decodifica: TABLIST
    tb_codlist (smallint) — codice listino
    tb_deslist (varchar 30) — descrizione
    tb_ivato   (varchar 1) — flag IVA inclusa

SCAGLIONI QUANTITA:
  Ogni riga listini copre un range: lc_daquant..lc_aquant (BETWEEN).
  Se lc_daquant=0 e lc_aquant=9999999999 → prezzo valido per qualsiasi quantita.
  Esempio pratico:
    Articolo X, fornitore Y:
      1-100 PZ   → prezzo 5.00
      101-500 PZ → prezzo 4.50
      501+   PZ  → prezzo 4.00

VALIDITA TEMPORALE:
  Ogni riga ha finestra: lc_datagg..lc_datscad (BETWEEN con data ordine).
  Quando un listino scade, ne viene creato uno nuovo con date aggiornate.

CASCATA DI RICERCA PREZZO (logica BCube — CercaPrezzo in CLDLBMENU.cs):
  BCube cerca il prezzo in 4 step con fallback:

  STEP 1 — CercaPrezzo_PreList: prezzo di listino base (lc_conto=0, listino generico)
    SELECT TOP 1 FROM listini WHERE lc_codart=@art AND lc_conto=0
      AND lc_listino=@listino AND @data BETWEEN lc_datagg AND lc_datscad
      AND @qta BETWEEN lc_daquant AND lc_aquant
    ORDER BY lc_datagg DESC, lc_daquant

  STEP 2 — CercaPrezzo_Listino: prezzo specifico, privilegia il fornitore
    SELECT FROM listini WHERE lc_codart=@art
      AND ((lc_conto=@fornitore AND lc_listino=0)    -- prezzo dedicato fornitore
           OR (lc_conto=0 AND lc_listino=@listino))  -- prezzo generico listino
      AND @data BETWEEN lc_datagg AND lc_datscad
      AND @qta BETWEEN lc_daquant AND lc_aquant
    ORDER BY lc_conto DESC, lc_codtpro DESC, lc_datagg DESC, lc_daquant
    → lc_conto>0 ha priorita massima (prezzo fornitore specifico batte generico)

  STEP 3 — CercaPrezzo_Costi: fallback su artprox (costi interni)
    Listini speciali (negativi):
      0  = ultimo costo (artprox)
      -1 = costo medio (artprox)
      -2 = ultimo prezzo d'acquisto medio (artprox) — DEFAULT per proposte MRP
      -3 = prezzo FIFO
      -4 = costo medio ponderato

  STEP 4 — CercaPrezzo_Valuta: se serve valuta estera (lc_codvalu>0)

SCELTA DEL LISTINO PER PROPOSTE MRP (CLEDBEMRP.cs):
  Per ol_tipork="O" (acquisto — il nostro caso):
    - Se fornitore specificato (lConto!=0): usa anagra.an_listino (listino fornitore)
    - Se fornitore=0: usa impostazione BSDBEMRP > Listino_Default_POF (default=-2)

COMPORTAMENTO REALE QUANDO an_listino = 0 (verificato su sorgenti decompilati):
  BCube NON ha fallback intelligente. Trace esatto:
    CLEDBEMRP.cs:11890 → nListino = NTSCInt(dataRow["an_listino"])  // copia diretta, NULL→0
    Tra linea 11890 e 12106 NON esiste alcun controllo "if nListino == 0".
    Il valore 0 viene passato letteralmente al motore prezzi.

  Con nListino=0, il motore prezzi (CLDLBMENU.cs:7127-7146):
    1. Cerca prezzo specifico fornitore (lc_conto=@fornitore AND lc_listino=0) → OK
    2. Se non trova → fallback su artprox (CercaPrezzo_ArtProx, ultimo costo)
    3. NON consulta altri listini generici, tabparam, gruppi merceologici, o categorie

  Il setting Listino_Default_POF (da ARCPROC.dbo.regprop) scatta SOLO quando
  lConto==0 (fornitore mancante), NON quando il fornitore esiste con an_listino=0.
  In piu, la config UJET11 ha BSVEBOLL/Tipo_list_costi_mat_carichi = "3" che
  bypassa completamente la lettura di Listino_Default_POF e forza nListinoPOF = -2.

  Conseguenza: an_listino=0 per la maggior parte dei fornitori e BY DESIGN.
  Quei fornitori usano prezzi specifici (lc_conto=@fornitore), non listini generici.

DISTRIBUZIONE REALE LISTINI (produzione BCUBE2, aprile 2026):
  Listini generici (lc_conto=0): esistono 15+ listini diversi.
    I piu grossi: listino 50 (8.982 righe), 8 (3.525), 51 (3.379), 99 (2.682).

TABELLA ARTPROX — FALLBACK ULTIMO COSTO:
  Quando il motore prezzi non trova nulla in listini, ricade su artprox.

================================================================================
26. TRAPPOLA: "ANNULLA + RIEMETTI" NON PRESERVA IL NUMERO ORDINE
================================================================================

CONTESTO:
  Per implementare il flusso "Unisci articoli a ordine pending" (merge) era stato
  ipotizzato di chiamare /annulla-ordine seguito da /emetti-ordine con gli articoli
  vecchi+nuovi, partendo dall'assunto che il numord venisse riutilizzato dal
  restore di tabnuma. ASSUNTO FALSO.

PERCHE NON FUNZIONA:
  /annulla-ordine (routes/gb2/ordini.js:328-342) decrementa tb_numprog SOLO se
  l'ordine cancellato e' letteralmente l'ultimo progressivo della serie/anno:

    if (tabnuma.tb_numprog === numordInt) {
        UPDATE tabnuma SET tb_numprog = tb_numprog - 1
    }

  In ambiente multi-operatore questo non e' mai garantito: fra il momento in cui
  il nostro pending e' stato emesso e il momento del merge, un altro operatore
  (o lo stesso su un altro fornitore) puo' aver bruciato il progressivo
  successivo. In quel caso l'annullo lascia un BUCO permanente nella numerazione
  e il "merge" genera un ordine con numord diverso da quello originale.

  E' lo stesso vincolo di BCube: anche la DelNuma dell'ERP recupera il numero
  solo se cancelli l'ultimo. Non e' un limite della nostra SP — e' la regola
  del gestionale.

SOLUZIONE IMPLEMENTATA:
  SP usp_AggiungiRigheOrdineFornitore in [GB2_SP]:
    - INSERT nuove righe in movord (mo_riga continua da MAX+1)
    - UPDATE testord ricalcolando totali + castelletto IVA
    - Refresh saldi BCube (keyord/artpro/artprox/lotcpro) solo per righe nuove
    - Applock per-ordine 'ModOrd_{anno}_{serie}_{numord}'
    - Duplicati codart CONSENTITI: ogni riga e' indipendente in movord
      (BCube supporta nativamente piu righe dello stesso articolo)
  Endpoint: POST /modifica-ordine

  SP usp_RimuoviRigaOrdineFornitore in [GB2_SP]:
    - Operazione inversa: rimuove UNA riga da movord
    - Sottrae delta da keyord/artpro/artprox/lotcpro
    - Ricalcola totali testord + castelletto IVA
    - Se ultima riga → cancella ordine + ripristina tabnuma
    - Aggiorna ordini_emessi + SnapshotProposte (Gestita=0) per far ricomparire la proposta
  Endpoint: POST /rimuovi-riga-ordine

VERIFICA PRECONDIZIONI:
  - Ordine deve esistere, td_flevas='N', nessuna riga evasa/prenotata.
  - email_inviata deve essere 0 (verificato dal dialog merge lato client).
  - Merge proposto solo per ordini con STESSA data consegna.

STATUS: Plan "gestione-lista-ordini" (aprile 2026) inizialmente lo descriveva in
modo errato. Corretto: il flusso merge deve usare la modifica in-place, mai
annulla+riemetti.
