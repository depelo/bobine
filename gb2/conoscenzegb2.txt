================================================================================
1. CONTESTO E POSIZIONAMENTO NELL'ECOSISTEMA
================================================================================

GB2 (Gabriele 2.0) e l'app MRP (Material Requirements Planning) di U.Jet s.r.l.
Sostituisce un vecchio programma Access VBA collegato al database ERP BCube.

GB2 e integrata nell'ecosistema PortalUjet come modulo dipartimentale,
seguendo il paradigma "1 App = 1 Database = 1 Visto":
  - Database silos: [GB2] su 192.168.0.163
  - Tabella Visto: [GB2].[dbo].[Operators] (IDOperator, IDUser, Admin, IsActive)
  - Registrata in [GA].[dbo].[Modules] come IDModule = 4, TargetDb = 'GB2'
  - Ruoli: Admin (UAD = Ufficio Admin) e Base (UAC = Ufficio Acquisti)
  - ENTRAMBI i ruoli richiedono password (diverso da Bobine dove Base no)

Il Captain Console gestisce l'assegnazione dei visti (permessi) agli utenti.
Dopo l'assegnazione, l'utente vede "Gabriele 2.0" nel menu laterale.

NOTA IMPORTANTE: Il file conoscenze.txt nella root del progetto bobine contiene
le regole architetturali generali di PortalUjet (Passaporto/Visto, Zero-Trust,
pool isolati, ecc.). Quelle regole si applicano anche a GB2. Questo documento
copre solo le specificita dell'app GB2/MRP.

2. ARCHITETTURA TECNICA
================================================================================

Stack:
  - Backend: Node.js + Express, integrato nel server.js principale (porta 443)
  - Database: SQL Server 2022 su 192.168.0.163
  - Frontend: SPA vanilla (HTML + CSS + JS, NO framework)
  - PDF: PDFKit (server-side)
  - Email: Nodemailer
  - Libreria DB: mssql (node-mssql)
  - Dev locale: server.js + PM2 (stessi file di produzione)
  - Crittografia: AES-256-GCM (config/crypto.js)

Server:
  - 192.168.0.163 (PC-SVILUPPO): server dove gira l'app (server.js/PM2, porta 443).
    Ospita i database: MRP, GB2, GA, BOB, CAP, PE, e una COPIA di UJET11 (solo prova).
  - bcube2: server ERP di produzione. Ospita il VERO database UJET11 con i dati
    reali usati dai 40 dipendenti. L'app NON si connette direttamente a bcube2.

--- MAPPA DEI DATABASE E COME SI RAGGIUNGONO ---

  Ci sono due categorie di dati, ognuna con il suo percorso di accesso:

  A) DATI ERP (tabelle UJET11: artico, anagra, movord, ordlist, testord, ecc.)
     Questi dati vivono su BCUBE2 in produzione. L'app non si collega mai
     direttamente a BCUBE2. Invece:
     - IN PRODUZIONE: il pool si connette a MRP@163 dove ci sono 21 viste
       che fanno da ponte trasparente (SELECT * FROM [BCUBE2].[UJET11].[dbo].[...]).
       Le query vedono le stesse tabelle come se fossero locali.
     - IN PROVA: il pool si connette DIRETTAMENTE a UJET11 sul server di prova
       (es. UJET11@163 che e una copia, o altri server configurati dall'operatore).
       Le viste NON vengono usate. Le stesse query funzionano perche i nomi
       delle tabelle sono identici.
     Il codice non deve preoccuparsi di questa differenza: usa getPoolMRP(userId)
     e il pool giusto viene restituito automaticamente.

  B) DATI LOCALI SU 163 (sempre li, sia in produzione che in prova)
     Tre database fisici su 192.168.0.163 che NON cambiano mai a seconda del modo:
     - MRP: tabelle custom (ordini_emessi, Politica, ScMin, VarScMin),
            stored procedures (usp_CreaOrdineFornitore + varianti _T{id}),
            e le 21 viste ponte verso BCUBE2.
     - GB2:  silos dell'app (Operators, UserPreferences, TestProfiles,
            ElaborazioniMRP, SnapshotProposte).
     - Analisi_scorte (su bcube2, via vista Riep in MRP): tabella ETL di Fabrizio.
     Questi si raggiungono SEMPRE con getPoolProd(), che punta a MRP@163.
     Per GB2 si usa naming cross-database: [GB2].[dbo].[NomeTabella].

--- REGOLA DECISIONALE PER SCEGLIERE IL POOL (CRITICA) ---

  Quando scrivi una nuova query, chiediti:
  "Sto leggendo/scrivendo dati ERP che in produzione vivono su BCUBE2?"

  SI (artico, movord, ordlist, anagra, testord, movmag, artpro, ecc.)
    → usa getPoolMRP(userId)
    In produzione passera dalle viste MRP@163 → BCUBE2.
    In prova andra diretto su UJET11@server_prova.

  NO (ordini_emessi, SP, [GB2].*, ElaborazioniMRP, SnapshotProposte, ecc.)
    → usa getPoolProd()
    Punta SEMPRE a MRP@163 indipendentemente dal modo dell'operatore.
    Per tabelle GB2 usa naming cross-database [GB2].[dbo].[...].

  CASO SPECIALE — Riep:
    → usa getPoolRiep(userId)
    In produzione: passa dalla vista in MRP (→ BCUBE2.Analisi_scorte).
    In prova CON Riep locale: usa il pool prova.
    In prova SENZA Riep locale: fallback a getPoolProd() (dati reali, sola lettura).

  ERRORE TIPICO DA EVITARE:
    Usare getPoolMRP(userId) per ordini_emessi o tabelle GB2. In prova,
    getPoolMRP restituisce il pool verso UJET11@server_prova dove quelle
    tabelle NON esistono → errore a runtime. Queste tabelle stanno su MRP@163
    e si raggiungono SOLO con getPoolProd().

--- GESTIONE CONNESSIONI — POOL PER UTENTE ---

  config/db-gb2.js gestisce i pool con isolamento per operatore:

  poolProd (singleton condiviso):
    - Pool fisso verso MRP su 192.168.0.163
    - MAX 30 connessioni (condiviso tra tutti gli operatori in produzione)
    - Usato anche per SP, GB2, TestProfiles (sempre su 163)
    - Creato lazy al primo utilizzo, vive per tutta la sessione del server

  userStates (Map<userId, UserState>):
    - Ogni operatore ha il suo stato: pool, profilo attivo, hasRiep, lastActive
    - Il pool di prova e INDIVIDUALE: lo switch di Pablo non impatta Marco
    - MAX 5 connessioni per pool di prova (1 operatore alla volta)
    - Pool chiuso automaticamente dopo 30 minuti di inattivita (cleanup ogni 5 min)
    - Con 8 operatori simultanei: max 1 pool prod (30 conn) + 8 pool prova (5 conn ciascuno)

  Funzioni esportate (tutte accettano userId):
    getPoolMRP(userId)       — pool DATI ERP: produzione (→ viste MRP) o prova (→ UJET11 diretto)
    getPoolProd()            — pool DATI LOCALI 163: MRP, GB2, SP (SEMPRE lo stesso)
    getPoolRiep(userId)      — pool per Riep con fallback a produzione se manca in prova
    getActiveProfile(userId) — profilo attivo per l'utente
    isProduction(userId)     — true se l'utente e in produzione
    switchToTest(userId, p)  — crea pool prova individuale per l'utente
    switchToProduction(userId) — chiude pool prova, torna a produzione

  Le route ottengono userId da getUserId(req) e lo passano a tutte le funzioni pool.
  In dev mode (skipAuth), userId = 0 (default).

  config/db.js (principale): pool per GA, PE, ET, BOB (NON usato da GB2)
  Le rotte GB2 usano SOLO db-gb2.js, MAI db.js

--- LISTA COMPLETA DELLE 21 VISTE MRP (ponte verso BCUBE2, SOLO PRODUZIONE) ---

  artico, anagra, artfasi, artpro, artmaga, artdef, artdefx,
  movord, movdis, movmag, ordlist, testord,
  tabmaga, tabnuma, tabciva, tabpaga,
  listprod, zzassdisp,
  __MOTIPORK, __MMTIPORK,
  Riep (punta a BCUBE2.Analisi_scorte.dbo.Riep — tabella ETL di Fabrizio)

  REGOLA INVIOLABILE: le viste devono SEMPRE puntare a [BCUBE2].[UJET11].[dbo].*
  (cross-server). NON modificarle mai per profili di prova — la prova non usa le viste.

Ordine di caricamento script (in gb2.html, fondo body):

  4. gb2-parametri.js
  5. gb2-progressivi.js
  6. gb2-proposta.js

  DOMContentLoaded -> MrpApp.init() -> MrpTheme.init() -> MrpDbConfig.init() -> MrpParametri.init()
  securityReady (da sicurezza.js) -> gb2-bootstrap -> buildDynamicMenu()


================================================================================
7. FLUSSO MRP — LOGICA DI BUSINESS
================================================================================

Elaborazione per singolo articolo (funzione caricaMRP):
  1. Legge giacenza base da artpro (ap_esist, ap_ordin, ap_impeg) — usa il pool passato come parametro
  2. Legge ordini fornitore aperti da movord (tipork='O', serie='F')
  3. Legge ordini MRP da ordlist (tipork='O','H','Y')
  4. Legge emissioni gia fatte da ordini_emessi — usa SEMPRE getPoolProd() perche'
     ordini_emessi risiede esclusivamente nel database MRP su 163, mai in UJET11 di prova.
     BUG FIX (2026-04-08): prima usava getUserId(req) dove req non esisteva nel contesto
     della funzione, causando il fallimento silenzioso della query (try/catch vuoto).
  5. Calcola progressivo per magazzino/fase con disponibilita
  6. Gestisce articoli esauriti con sostitutivo (blocchi esaurimento/sostitutivo/combinato)
  7. Restituisce array di righe tipizzate: padre, magazzino, totale, figlio, ecc.

REGOLA: caricaMRP riceve il pool come parametro (puo' essere produzione o prova per le

8. FLUSSO EMISSIONE ORDINI FORNITORE
================================================================================

Passo 1 — Conferma (client-side):
  L'operatore conferma quantita, data, prezzo nel pannello decisionale.
  Dati in MrpApp.state.ordiniConfermati. NIENTE va in DB in questa fase.

Passo 2 — Emissione (POST /emetti-ordine):
  Chiama SP usp_CreaOrdineFornitore (o usp_CreaOrdineFornitore_T{id} in prova).
  La SP usa il placeholder {{UJET11_REF}} risolto al deploy:
  - Produzione: [BCUBE2].[UJET11].[dbo] (scrive su UJET11 di produzione)
  - Prova T3:   [SERVER_PROVA].[UJET11].[dbo] (scrive su UJET11 di prova)
  La SP:
  a) Parsa JSON articoli con OPENJSON
  b) Arricchisce da artico (descrizione, IVA)
  c) Calcola castelletto IVA
  d) LOCK applicativo (sp_getapplock) per serializzare numerazione
  e) Legge numord da tabnuma (fonte autoritativa BCube) e la aggiorna
  f) INSERT testord + movord su UJET11
  g) INSERT ordini_emessi (registro locale)
  h) Restituisce testata + righe per il PDF

Passo 2b — Post-emissione (dopo successo SP, in Node.js):
  a) UPDATE ordini_emessi SET ambiente = 'produzione'|'prova' (via getPoolProd)
  b) UPDATE SnapshotProposte SET Gestita=1, OrdineEmessoID=id (via poolGB2)
  c) UPDATE ElaborazioniMRP SET TotaleGestite = COUNT(...) (via poolGB2)

Passo 2c — Controllo duplicati (pre-emissione):
  POST /controlla-duplicato verifica se esistono gia ordini per stessi
  articoli+fornitore+elaborazione in ordini_emessi. Se si, mostra warning
  con opzione Procedi/Annulla.

Passo 3 — PDF generato server-side con PDFKit, ritornato come base64.

Passo 4 — UI: ri-fetch dati dal server (data-driven, mai manipolazione DOM diretta).

Passo 5 — Email (opzionale): POST /invia-ordine-email con PDF allegato.
  Usa la configurazione SMTP dell'operatore loggato (da UserPreferences).
  Dopo invio: UPDATE ordini_emessi SET email_inviata=1, email_inviata_il=GETDATE().

Passo 6 — Storico (modale):
  Bottone "Storico" nella header proposta apre modale con lista ordini emessi.
  Filtri: elaborazione corrente (checkbox), date da/a, fornitore.
  Azioni per riga: Visualizza (riapre modale risultato), PDF, Re-invia email.

16. SISTEMA TRACCIAMENTO ELABORAZIONI MRP
================================================================================

L'elaborazione MRP notturna di BCube genera le proposte in ordlist. L'app GB2
deve sapere a quale "batch" appartengono le proposte correnti per:
- Distinguere proposte nuove da quelle gia gestite
- Permettere lo storico degli ordini emessi per elaborazione
- Evitare ri-emissioni accidentali

FINGERPRINT:
  Il campo ol_ultagg di ordlist contiene la data dell'ultima elaborazione MRP.
  Tutte le proposte di un batch condividono lo stesso ol_ultagg. L'app rileva
  il fingerprint con:
    SELECT TOP 1 ol_ultagg AS fingerprint, COUNT(*) AS cnt
    FROM dbo.ordlist WHERE ol_tipork='O'
    GROUP BY ol_ultagg ORDER BY cnt DESC

RILEVAZIONE (in GET /proposta-ordini):
  1. Query fingerprint (su pool MRP)
  2. Check ElaborazioniMRP per (Fingerprint, Ambiente) su poolGB2 = getPoolProd()
  3. Se non esiste: INSERT in ElaborazioniMRP + bulk INSERT in SnapshotProposte
     (chunked a 100 righe per batch, parametri $1..$N)
     Pattern INSERT-or-SELECT: catch errore duplicate key (2601/2627) → SELECT
  4. Se esiste: riconciliazione — query ordini_emessi per elaborazione_id,
     UPDATE SnapshotProposte.Gestita=1, UPDATE TotaleGestite

RISPOSTA API (cambio formato):
  GET /proposta-ordini ora restituisce:
    { elaborazione: { id, fingerprint, totaleProposte, totaleGestite }, righe: [...] }
  Il frontend estrae elaborazione e salva in MrpApp.state.

FRONTEND — 5 SCENARI UX:
  1. Blocco ri-emissione: ordine gia emesso → badge "Emesso" + bottone "Visualizza Ordine"
     Click → GET /ordine-dettaglio → riapre modale risultato con PDF ed email
  2. Storico ordini: modale con lista ordini raggruppati per documento
     Filtri: elaborazione corrente, date, fornitore. Azioni: PDF, re-invio email
  3. Warning duplicati: prima di emettere, POST /controlla-duplicato verifica
     se stessi articoli+fornitore gia emessi in questa elaborazione
  4. Badge email: icona inviata/non-inviata per ogni ordine emesso
     Testo bottone: "Invia" vs "Re-invia" in base a email_inviata
  5. Proposte gestite in verde: classe .proposta-art-gestita (bordo+sfondo verde)
     se r.emesso && r.elaborazione_id === MrpApp.state.elaborazioneId

BANNER ELABORAZIONE:
  Riga info dopo le stats: "#ID | Batch MRP: dd/mm/yyyy HH:mm | Gestite: X/Y"

CAMBIO ELABORAZIONE:
  Se il fingerprint cambia (nuova elaborazione MRP notturna), il frontend svuota
  MrpApp.state.ordiniConfermati per evitare ordini basati su proposte obsolete.

POOL — vedi regola decisionale in sezione 2. In sintesi per questa sezione:
  - ElaborazioniMRP, SnapshotProposte, ordini_emessi → getPoolProd() (dati locali 163)
  - ordlist (fingerprint) → getPoolMRP(userId) (dati ERP, segue il modo dell'utente)

AMBIENTE:
  Ogni riga in ordini_emessi ha una colonna 'ambiente' ('produzione' o 'prova').
  Le query di storico filtrano per ambiente corrente, cosi ordini di prova
  non appaiono in produzione e viceversa.
  ElaborazioniMRP ha UNIQUE index su (Fingerprint, Ambiente) per separare
  elaborazioni rilevate in produzione da quelle rilevate in prova.

FALLBACK QUERY:
  Le query su ordini_emessi usano try/catch: se le colonne email_inviata o
  ambiente non esistono ancora (pre-deploy), il catch esegue una query senza
  quelle colonne. Questo garantisce compatibilita durante il primo deploy.

  va usata come [Qta] per evitare "Invalid column name".

  Il tool MCP mssql-ujet11 (se presente) si connette al database UJET11, NON a GB2.
  Non supporta operazioni cross-database. Le tabelle create con il tool MCP finiscono
  in UJET11 invece che in GB2, causando errori silenziosi.
  PER QUALSIASI OPERAZIONE SUL DATABASE (create table, query, insert, alter, etc.)
  USARE SEMPRE Node.js diretto con uno script temporaneo, es:
    node -e "const sql = require('mssql'); ..." oppure uno script .js dedicato.
  Questo garantisce il controllo completo su quale database viene usato.
